# Feature: Revive Source Generator with IIncrementalGenerator

**Status:** APPROVED (v4.1 — gaps G1–G4, S2–S3 resolved)
**Work Type:** Feature
**Created:** 2026-03-15
**Updated:** 2026-03-15 (v4.1)

## Problem

The original Roslyn source generator (`KeyValueSourceGenerator : ISourceGenerator`) was removed in commit `7ceed95` (May 2024). The `[Key]` and `[StorageSerializable]` attributes exist in `src/Domain/Attributes/` but do nothing — no code consumes them. The `_Generated.cs` files in `src/Samples/` are hand-written files with misleading `<auto-generated>` headers.

**Note:** The attribute was originally named `[Serializable]`, which conflicts with `System.SerializableAttribute`. With `ImplicitUsings` enabled (which imports `System`), consumers cannot write `[Serializable]` without full qualification or a `using` alias. Since the attribute has zero external consumers today (it's dead code), this spec renames it to `[StorageSerializable]` at zero cost.

Consumers who want to use file-backed storage engines (`IStorageEngine`, `IBulkStorageEngine`, `ILsmStorageEngine`) must manually implement `ISerializable<T>.Serialize(Stream)`, `static T Deserialize(Stream)`, and `IComparable<T>.CompareTo(T)` for every key and value type. This is significant boilerplate for what should be a declarative operation.

## User Story

As a library consumer, I want to annotate my `partial record struct` types with `[Key]` or `[StorageSerializable]` and have the compiler generate `ISerializable<T>` (and `IComparable<T>` + `IEquatable<T>` for keys) implementations automatically, so I can focus on my domain types instead of writing serialization boilerplate.

## Acceptance Criteria

### Generator Core
1. A new `Boutquin.Storage.SourceGenerator` project exists, targeting `netstandard2.0`, using `IIncrementalGenerator`
2. Types annotated with `[Key]` get generated `ISerializable<T>` + `IComparable<T>` + `IEquatable<T>` implementations, plus `==`, `!=`, `<`, `>`, `<=`, `>=` operators
3. Types annotated with `[StorageSerializable]` get generated `ISerializable<T>` implementations (no `IComparable`, no `IEquatable`)
4. Generated serialization uses `BinaryWriter`/`BinaryReader` with `leaveOpen: true` (matching existing `SerializableWrapper` and sample patterns)
5. Supported primitive property types: `int`, `long`, `string`, `float`, `double`, `bool`, `byte`, `char` (same set as `SerializableWrapper<T>`)
6. Nested `[StorageSerializable]`/`[Key]`-annotated types serialize recursively (e.g., `City` containing `IEnumerable<Attraction>`)
7. `IEnumerable<T>` collection properties serialize as count-prefixed sequences (matching existing `City_Generated.cs` pattern)
8. `CompareTo` for `[Key]` types uses `ComparisonHelper.SafeCompareTo` for the first property; multi-property keys use chained lexicographic comparison
9. Unsupported property types produce a compiler diagnostic (error, not silent failure)

### Generated Code Quality

16. Generated code uses LF line endings (matching Verify defaults and `.editorconfig` if present)

## Design Decisions

### Why IIncrementalGenerator (not ISourceGenerator)

`ISourceGenerator` is the V1 API — it re-runs on every keystroke and doesn't cache. `IIncrementalGenerator` (V2) uses a pipeline model with caching, making it dramatically faster in IDE scenarios. Microsoft recommends V2 for all new generators. Both StronglyTypedId and Vogen use V2.

### Pipeline architecture (borrowed from StronglyTypedId + Vogen)

The generator uses `ForAttributeWithMetadataName` — the most performant Roslyn attribute discovery method. Both StronglyTypedId and Vogen use this over `CreateSyntaxProvider` with manual attribute checking because it short-circuits entirely when no matching attributes exist in the compilation.

**Critical caching constraint:** The pipeline must never combine with raw `CompilationProvider`. The `ForAttributeWithMetadataName` transform receives `SemanticModel` in its callback — extract all needed data there. Assembly-level defaults are extracted via their own `ForAttributeWithMetadataName` pipeline, not via `CompilationProvider`. Combining with `CompilationProvider` invalidates caching on every keystroke (Andrew Lock, Part 9).

**No Roslyn types in pipeline output:** Pipeline intermediate types must never hold `SyntaxNode`, `ISymbol`, `Compilation`, or `Location` references — these are not equatable across compilations and defeat incremental caching. All data must be extracted into plain value types in the transform step.

Key pipeline infrastructure types:

- **`EquatableArray<T>`** — `readonly struct` wrapping `T[]`, implementing `IEquatable<EquatableArray<T>>`, `IEnumerable<T>`. Equality via `SequenceEqual`. Custom `GetHashCode` combining element hashes via XOR + prime multiplication (not `HashCode.Combine` — that's unavailable on netstandard2.0 where the generator itself runs). Borrowed from StronglyTypedId.

- **`DiagnosticInfo`** — Record with `Id`, `Message`, `Severity`, and a custom `LocationInfo` record struct containing `string FilePath`, `TextSpan Span`, `LinePositionSpan LineSpan` (not a `Microsoft.CodeAnalysis.Location` reference — `Location` is not equatable across compilations). Factory method `DiagnosticInfo.Create(Diagnostic)` extracts serializable data. Method `ToDiagnostic()` reconstructs for reporting. Borrowed from StronglyTypedId.

- **`TypeToGenerate`** — Record struct capturing all data extracted from the semantic model: `Name`, `Namespace` (nullable for global namespace), `Accessibility` (public/internal), `IsKey` (bool), `Properties` (`EquatableArray<PropertyInfo>`), `ParentTypes` (`EquatableArray<ParentTypeInfo>` for nested type support). All fields use equatable types for caching. The generator never re-reads the semantic model if this value hasn't changed.

- **`PropertyInfo`** — Record struct: `Name` (string), `TypeName` (string), `TypeKind` (enum: Primitive, AnnotatedType, PrimitiveCollection, AnnotatedCollection, Unsupported), `PrimitiveKind` (enum for the 8 supported primitives, nullable). Classifies properties at transform time so `CodeGenerator` never needs the semantic model.

- **`ParentTypeInfo`** — Record struct: `Name` (string), `Kind` (class/struct/record), `Modifiers` (string), `GenericParameters` (string?, e.g., `"<T, U>"`), `Constraints` (string?, e.g., `"where T : class"`). Supports generic parent types.

### Why BinaryWriter/BinaryReader (not BinaryPrimitives)

The CLAUDE.md convention says "explicit endianness — on-disk formats use `BinaryPrimitives` with little-endian." However, this applies to the **library's internal** on-disk formats (WAL, SSTable). The `ISerializable<T>` interface is a **consumer-facing contract** — it takes a `Stream` and consumers implement serialization however they want. The existing `SerializableWrapper`, all sample `_Generated.cs` files, and the `BinaryEntrySerializer` all use `BinaryWriter`/`BinaryReader`. The generator should match these established patterns.

### Why netstandard2.0 for the generator project

Roslyn source generators must target `netstandard2.0` — this is a hard requirement from the compiler hosting model. The generator runs inside the compiler process, which loads analyzers as `netstandard2.0` assemblies. Use the `Polyfill` NuGet package to backfill nullable annotations, `IsExternalInit` (required for `record` / `init` in generator code), and other modern APIs (as StronglyTypedId does). `LangVersion 14.0` is set because C# language features (records, pattern matching, raw strings) are compiler features that work regardless of target framework — `Polyfill` bridges any missing runtime types.

### Why Microsoft.CodeAnalysis.CSharp 4.4.0 (not higher)

`ForAttributeWithMetadataName` was introduced in Roslyn 4.4.0 (shipped with .NET 7 SDK). Using 4.4.0 as the minimum ensures compatibility with consumers using .NET 7+ SDKs. StronglyTypedId also uses 4.4.0 as its floor. Higher versions (4.8+ for .NET 8, 4.12+ for .NET 9) would unnecessarily restrict SDK compatibility. If a specific API from a newer Roslyn version is needed during implementation, the version can be bumped with a documented justification.

### Attribute rename: `[Serializable]` → `[StorageSerializable]`

`System.SerializableAttribute` exists in `System` which is auto-imported via `ImplicitUsings`. The existing `City.cs` and `Attraction.cs` in Samples already must use `[Domain.Attributes.Serializable]` to disambiguate. Renaming to `[StorageSerializable]` eliminates the conflict. The attribute has zero external consumers (it's dead code), so this is a zero-cost rename.

### Attribute location: Domain project (no change)

The `[Key]` and `[StorageSerializable]` attributes stay in `Boutquin.Storage.Domain`. The generator does **not** reference Domain as a project dependency — it discovers attributes by fully-qualified metadata name string via `ForAttributeWithMetadataName`. The *consumer* project references Domain (to get the attributes) and the generator (as an analyzer). This is the standard pattern used by both StronglyTypedId and Vogen: attributes in the main library, generator in a separate standalone analyzer assembly with no dependency on the attribute assembly.

### `[Conditional]` on attributes (borrowed from StronglyTypedId)

Add `[Conditional("BOUTQUIN_STORAGE_GENERATOR")]` to both `KeyAttribute` and `StorageSerializableAttribute`. This causes the attributes to vanish from compiled output (they're only needed at compile time for the generator). StronglyTypedId uses this pattern to keep consumer assemblies clean.

**Note:** `ForAttributeWithMetadataName` operates on syntax (source code), not IL (post-emit), so `[Conditional]` does not prevent the generator from finding annotated types. The generator sees the attribute in source regardless of whether it would be emitted to the assembly.

### Assembly-level defaults attribute (borrowed from both)

Add `[StorageDefaults]` assembly-level attribute for project-wide generator configuration. Both StronglyTypedId (`[StronglyTypedIdDefaults]`) and Vogen (`[VogenDefaults]`) provide this. Initially the only configurable option is whether to generate comparison operators for `[Key]` types (default: yes). This is the extension point for future configuration.

### Equality and comparison generation (borrowed from Vogen)

For `[Key]` types, generate:
- `IEquatable<T>` with `Equals(T)`, `Equals(object)`, `GetHashCode()`
- `==` and `!=` operators
- `IComparable<T>` with `CompareTo(T)`
- `<`, `>`, `<=`, `>=` operators (Vogen generates these alongside `IComparable`)

`GetHashCode` uses `EqualityComparer<T>.Default.GetHashCode(Value)` for struct types (matching Vogen's pattern). Multi-property keys combine hashes via `HashCode.Combine()`.

**Record struct coexistence:** When a `partial record struct` provides `Equals(T)`, `GetHashCode()`, `==`, and `!=`, the compiler skips synthesizing its own versions. The C# spec states: "The synthesized `operator ==` / `operator !=` is not generated when a user-defined version with matching signature exists." The generated and synthesized equality will agree because both operate on the same properties. A dedicated `RecordStructSynthesisCoexistence` test verifies no CS0111 (duplicate member) errors.

### `global::` prefixes (borrowed from Vogen)

All type references in generated code use `global::` prefixes (e.g., `global::System.IO.Stream`, `global::Boutquin.Storage.Domain.Helpers.ComparisonHelper`). This prevents namespace conflicts when consumer code has types with the same names as framework types. Vogen applies this consistently throughout all generated code.

### Diagnostic suppressors (borrowed from Vogen)

Include a `DiagnosticSuppressor` (extends `DiagnosticSuppressor`, not `DiagnosticAnalyzer`) that suppresses diagnostics which are false positives for generated types. Reports supported suppressions via the `SupportedSuppressions` property. Vogen includes suppressors for similar scenarios.

Suppressed diagnostics:
- **CA1036** ("Override methods on comparable types") — the generator already generates the required operator overloads
- **S1210** (SonarAnalyzer: "When implementing IComparable, you should also override ==, !=, <, <=, >, >=") — SonarQube equivalent of CA1036

**Implementation:** The suppressor uses `Compilation.GetTypeByMetadataName("Boutquin.Storage.Domain.Attributes.KeyAttribute")` to obtain the attribute symbol, then checks `INamedTypeSymbol.GetAttributes()` on each diagnostic's target type to verify it has `[Key]`.

### Code generation strategy

Use `StringBuilder` for assembling generated source text (matching StronglyTypedId's pattern). Reuse a single `StringBuilder` per type generation to reduce allocations. Avoid `IndentedTextWriter` (requires `System.CodeDom` dependency). Avoid raw string interpolation for large outputs — it forces string realization on every append. The `CodeGenerator` class exposes a `Generate(TypeToGenerate)` → `string` method that builds the full source text.

### Minimum consumer TFM: .NET 7+

The `ISerializable<T>` interface uses `static abstract T Deserialize(Stream)`, which requires C# 11 / .NET 7+. This means `HashCode.Combine()`, `using var`, and all other modern APIs are available in generated code. The generator itself targets netstandard2.0, but generated code runs in the consumer's TFM (net7.0+).

### `[Conditional]` on attributes — how it actually works

Unlike StronglyTypedId (which generates attribute source via `RegisterPostInitializationOutput` so attributes are emitted as generated source), our attributes live in the Domain assembly. `[Conditional("BOUTQUIN_STORAGE_GENERATOR")]` uses a constant that is intentionally never defined — the attributes always vanish from consumer IL. This is fine because `ForAttributeWithMetadataName` reads the semantic model built from source, not emitted IL. The constant name is intentionally obscure to prevent accidental definition.

### Composite key comparison

For `[Key]` types with a single property, `CompareTo` delegates to `ComparisonHelper.SafeCompareTo(this.Prop, other.Prop)`. For multi-property keys, the generator produces chained comparisons (compare first property, if equal compare second, etc.) — standard lexicographic ordering.

4. Create `src/SourceGenerator/StorageSourceGenerator.cs` implementing `IIncrementalGenerator`:

   **Pipeline setup (`Initialize`):**
   ```
   a. ForAttributeWithMetadataName("Boutquin.Storage.Domain.Attributes.KeyAttribute")
      → Transform to TypeToGenerate (with IsKey = true)
   b. ForAttributeWithMetadataName("Boutquin.Storage.Domain.Attributes.StorageSerializableAttribute")
      → Transform to TypeToGenerate (with IsKey = false)
   c. ForAttributeWithMetadataName("Boutquin.Storage.Domain.Attributes.StorageDefaultsAttribute")
      → Transform to defaults config
   d. Combine all three providers
   e. RegisterSourceOutput → CodeGenerator.Generate()
   ```

   **Transform logic (shared for Key + StorageSerializable):**
   - Extract type name, containing namespace (null if global), file-scoped namespace support
   - Detect accessibility: `public` or `internal` from declared modifiers
   - Extract parent type hierarchy for nested types (name, kind, modifiers, generic parameters, constraints for each parent)
   - Extract all public properties: name, type symbol → classify into `PropertyInfo.TypeKind` (Primitive, AnnotatedType, PrimitiveCollection, AnnotatedCollection, Unsupported)
   - For `IEnumerable<T>` properties: determine element type — if primitive, set `TypeKind = PrimitiveCollection`; if `[StorageSerializable]`/`[Key]` annotated, set `TypeKind = AnnotatedCollection`; otherwise `Unsupported`
   - Validate: `partial` modifier present (else DiagnosticInfo for BSSG002), `record struct` (else DiagnosticInfo for BSSG003)
   - Check for dual-attribute: if type has both `[Key]` and `[StorageSerializable]`, emit DiagnosticInfo for BSSG005
   - Check for existing interface implementations: if type already declares `ISerializable<T>`, `IComparable<T>`, or `IEquatable<T>`, emit DiagnosticInfo for BSSG006 and skip those interfaces in generation
   - Return `(TypeToGenerate, EquatableArray<DiagnosticInfo>)`

5. Create `src/SourceGenerator/CodeGenerator.cs` — assembles generated source text:

   **Generated file structure:**

   // Copyright © 2024-present Pierre G. Boutquin. All rights reserved.
   // Licensed under the Apache License, Version 2.0.
   // See LICENSE file in the project root for full license information.
   //

   // This code was generated by Boutquin.Storage.SourceGenerator.
   // Do not edit this file manually.
   // </auto-generated>
   #nullable enable

   **Serialize(Stream):**
   ```csharp
   public void Serialize(global::System.IO.Stream stream)
   {
       using var writer = new global::System.IO.BinaryWriter(stream, global::System.Text.Encoding.UTF8, leaveOpen: true);
       writer.Write(Property1);           // primitive (value type — no null check needed)
       writer.Write(StringProp ?? throw new global::System.InvalidOperationException(
           "Property 'StringProp' cannot be null during serialization."));  // string — null guard
       Property2.Serialize(stream);       // nested [StorageSerializable]/[Key] type
       var collectionList = new global::System.Collections.Generic.List<T>(Collection);  // materialize once, no System.Linq dependency
       writer.Write(collectionList.Count);        // IEnumerable<T> count
       foreach (var item in collectionList)       // IEnumerable<T> elements
       {
           item.Serialize(stream);
       }
   }
   ```

   **String null guard rationale:** `BinaryWriter.Write(string)` throws `ArgumentNullException` on null. Although nullable properties are out of scope, `string` is a reference type — a consumer could construct `new MyValue(null!)`. The generated null guard produces a clearer error message than the framework's internal `ArgumentNullException`.

           collection.Add(T.Deserialize(stream));
       }
       return new <TypeName>(prop1, prop2, collection);
   }

## Scope Boundaries (NOT included)

- **BinaryPrimitives migration** — The generator uses `BinaryWriter`/`BinaryReader` (matching existing patterns). Migration to explicit endianness via `BinaryPrimitives` is a separate concern for internal library formats, not consumer-facing `ISerializable`.
- **NuGet packaging of the generator** — The generator is a project reference for now. Packaging it as a separate NuGet analyzer package (with `analyzers/dotnet/cs/` + `lib/` dual placement, `.targets` file) is a follow-up task. The spec documents the pattern (from StronglyTypedId and Vogen) but doesn't implement it.
- **Serialization converter generation** — Vogen generates System.Text.Json converters, EF Core ValueConverters, Dapper TypeHandlers, etc. This is valuable but out of scope for v1. The `[StorageDefaults]` attribute provides the extension point for future `Conversions` flags enum.
- **Array/List/Dictionary support** — Only `IEnumerable<T>` is supported (matching existing sample). Other collection types are future work.
- **Nullable property support** — Nullable value types (`int?`, `string?`) are out of scope. All properties are assumed non-nullable.
- **Custom serialization attributes** — No `[SerializeAs]`, `[Ignore]`, or similar attributes. The generator serializes all public properties of the record struct.
- **Generic attribute form** — Vogen uses `[ValueObject<T>]` (C# 11+). The current `[Key]` / `[StorageSerializable]` attributes don't take a type parameter and don't need one — the underlying type is the record struct itself. Not applicable here.
- **Validation / factory pattern** — Vogen enforces `From()` factory and blocks constructors. Not relevant for storage types where all property combinations are valid.
- **Code fix providers** — StronglyTypedId includes code fixes (e.g., add `partial` modifier). Deferred to follow-up.
- **Record class / class / struct support** — Only `record struct` is fully supported in v1. `record class` has reference semantics (nullable concerns, heap allocation) that complicate generated serialization and comparison code. BSSG003 warns on non-`record struct` types.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Generator discovers attributes by string name — attribute rename/move breaks silently | Low | High | Generator uses `ForAttributeWithMetadataName("Boutquin.Storage.Domain.Attributes.KeyAttribute")` etc. If attributes are renamed or moved, the generator silently stops finding them. Mitigated by snapshot tests that verify generated output for known inputs — a broken attribute name produces zero output, which fails the snapshot comparison. |
| netstandard2.0 constraint limits C# features in generator code | Medium | Low | Use `Polyfill` NuGet package to backfill nullable annotations and modern APIs. Avoid `static abstract` in generator code (only in generated output). |
| BinaryWriter endianness doesn't match BinaryPrimitives conventions | Low | Medium | This is intentional — `ISerializable` is consumer-facing, not internal format. Documented in Design Decisions. |
| Collection property detection (`IEnumerable<T>`) may be fragile | Medium | Medium | Use semantic model to check if type implements `IEnumerable<T>`. Test with `List<T>`, `T[]`, `IReadOnlyList<T>` in consumer tests. |
| Existing tests may directly reference the hand-written `_Generated.cs` types | Low | Low | Search for imports from Samples namespace in test files before deleting. |
| `[Conditional]` on attributes may break existing consumer code that reflects on attributes at runtime | Low | Medium | Document the behavioral change. Any consumer using reflection to find `[Key]`/`[StorageSerializable]` at runtime would need to define the `BOUTQUIN_STORAGE_GENERATOR` constant. This is rare — these attributes are compile-time markers. |
| Verify snapshot tests may be brittle across Roslyn version updates | Medium | Low | Pin `Microsoft.CodeAnalysis.CSharp` version. Update snapshots deliberately when upgrading. |
| `record struct` generates its own `Equals`/`GetHashCode` which may conflict with generator output | Medium | Medium | The generator implements `IEquatable<T>` explicitly. The record struct's synthesized members delegate to the same properties, so behavior is consistent. Test explicitly that generated and synthesized equality agree. |
